# AWS EKS Kubernetes Setup Guide

This guide details how to provision an Amazon Elastic Kubernetes Service (EKS) cluster and deploy your Makikibahay backend to it.

---

## 1. Prerequisites
Before you begin, you must have an AWS Account and these CLI tools installed on your local machine:
- **AWS CLI**: To manage AWS resources. ([Install Guide](https://docs.aws.amazon.com/cli/latest/userguide/getting-started-install.html))
- **eksctl**: The official CLI for Amazon EKS. ([Install Guide](https://eksctl.io/installation/))
- **kubectl**: Kubernetes command-line tool.

Once installed, authenticate your AWS CLI by running:
```bash
aws configure
```
Provide your AWS Access Key ID, Secret Access Key, and choose a region (e.g., `ap-southeast-1` for Singapore).

---

## 2. Provisioning the EKS Cluster
The fastest way to create a production-ready cluster is using `eksctl`.

1. Open your terminal and run:
   ```bash
   eksctl create cluster \
     --name makikibahay-cluster \
     --region ap-southeast-1 \
     --version 1.30 \
     --nodegroup-name standard-workers \
     --node-type t3.medium \
     --nodes 2 \
     --nodes-min 1 \
     --nodes-max 3 \
     --managed
   ```
   *(Note: This process takes about 15-20 minutes as it provisions VPCs, Security Groups, and EC2 instances).*

2. Verify the cluster is running:
   ```bash
   kubectl get nodes
   ```
   You should see your EC2 worker nodes listed as `Ready`.

---

## 3. Installing the NGINX Ingress Controller on AWS
AWS requires a Network Load Balancer (NLB) to securely route public internet traffic into your Kubernetes cluster. We will install the NGINX Ingress Controller, which automatically provisions an AWS NLB for you.

Run this exact command tailored for AWS:
```bash
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.8.2/deploy/static/provider/aws/deploy.yaml
```

Wait until the NLB is provisioned and the Ingress Controller pod is running:
```bash
kubectl get pods -n ingress-nginx
kubectl get svc -n ingress-nginx
```
Find the `EXTERNAL-IP` for the `ingress-nginx-controller` service. It will look something like `k8s-ingressn-XXXXXXXXX.elb.ap-southeast-1.amazonaws.com`.
**You must go to your Domain Registrar (e.g., Namecheap, GoDaddy, Route53) and create a `CNAME` record pointing `api.makikibahay.com` to this long AWS Load Balancer URL!**

---

## 4. Applying Your Manifests
Now that your cluster is running and your Ingress Controller is handling the Load Balancer, deploy your app:

1. Navigate to the root of your repository.
2. Edit `k8s/backend-secret.yaml` with your actual secure production secrets (e.g. your MongoDB Atlas connection string, JWT secret).
3. Apply the manifests:
   ```bash
   kubectl apply -f k8s/backend-config.yaml
   kubectl apply -f k8s/backend-secret.yaml
   # If you are using MongoDB Atlas (recommended), DO NOT apply mongo-statefulset.yaml.
   kubectl apply -f k8s/backend-deployment.yaml
   kubectl apply -f k8s/backend-service.yaml
   kubectl apply -f k8s/backend-ingress.yaml
   ```

Check if everything is running seamlessly:
```bash
kubectl get pods
```

## 5. Integrating with GitHub Actions (CI/CD)
To allow GitHub Actions to automatically deploy new updates when you push code, it needs access to your AWS cluster.

### Step A: Extract your Kubeconfig Context
To extract the raw Kubeconfig required for the GitHub Secret `KUBECONFIG` you created earlier:
```bash
cat ~/.kube/config
```
However, AWS `kubeconfig` uses short-lived tokens generated by the `aws` CLI locally. For GitHub Actions, you must use a dedicated IAM User methodology instead of long-lived kubeconfigs.

### Step B: The AWS Recommended CI/CD Methodology
Instead of passing `KUBECONFIG`, AWS recommends passing AWS Credentials to GitHub.
Modify your `.github/workflows/deploy-backend.yml` file to use `aws-actions/configure-aws-credentials` and `aws eks update-kubeconfig`.

In GitHub -> Settings -> Secrets, add:
- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY` (Ensure this IAM user has `eks:DescribeCluster` permissions).

Replace the **"Set up Kubectl"** step in your Action with:
```yaml
      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ap-southeast-1

      - name: Update Kubeconfig
        run: aws eks update-kubeconfig --name makikibahay-cluster --region ap-southeast-1
```
Now, whenever you push to main, GitHub Actions will securely assume the AWS identity, update its local cluster mapping, and deploy your new Docker containers!
